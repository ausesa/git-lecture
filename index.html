Chapter 1

Version Control Systems

Ask if class knows what it is.

Why do we need Version Control?

  • Safety
  • Collaboration
  • Maintenance
Ask class why they think it's needed.

Safety

If we make a mistake in our code and break everything, how do we fix it?

  • Copy and past folders to track versions?
    This is horrible! Imagine the hundreds of folders to manage.
    It only leads to confusion of where bugs were introduced and a bloated project folder.
    What if you couldn't find where you went wrong and had to redo half of the project?
  • Instead use a Version Control System
    Changes made to files are stored as differences between versions.
    When you've made a change to a file(s), save the difference it into the VCS.
Ever broken something and not known how to fix it?
If you use git you won't have to worry about it.

Collaboration

How do multiple people work on a project simultaneously?

  • Pass files around on a USB drive?
    No. This is the old way of sharing progress.
    Comparing two files and finding the difference between the two is very tedius and time consuming.
  • Instead use a Version Control System
    Changes made to files are stored as differences between versions.
    When you've made a change to code, save the difference it into the VCS.
This is how it used to be done, except on floppy drives.
Refer to GitHub and explain that we'll be using it later.

Maintenance

How do we support old versions of our code that have been deployed?

  • Fix current code set and re-deploy.
    More than likely, the programme has changed significantly and is now incompatible with the customer's system.
    The 'bug' could have been built into a new feature or is a dependency in something new.
  • Version control makes this a lot simpler. With version control you could find the bug, go back and fix it then deploy the changes to the customer.
Again, explain that examples will come in the next chapter.

Chapter 2

Git

What is Git?

Git is a very popular distributed version control system that you'll be using during this course.

  • What is a repository?
    A repo is your project code and history of changes that were made.
  • What goes in a repo?
    • Code
    • Documentation
    • The history of changes (this is managed is managed by git)
  • What doesn't go in a repo?
    • Compiled files (.exe .class .pyc)
    • Project setting files
git - the stupid content tracker
Why is it called git?
Torvalds said: "I'm an egotistical bastard, and I name all my projects after myself. First 'Linux', now 'git'."

> git init

  • Creates an empty repo
  • All history and settings are stored in the .git folder
Time to start the live demo.

> git status

  • Show us the status of the repo
    The branch we're on
    The files have changed
    The files are staged for commit
  • Git isn't automatic
    We need to tell it what files to track, stage and commit.

> git add <file> [...]

  • Adds a file to be tracked by git
    Simply adding a file to the repository does not mean its changes are being tracked by git.
  • Adding all the files in a new project can be tedious
    Fortunately, there is a shortcut.
  • > git add .
    This adds all files from the current working directory downwards.
    If run in the root directory of the project, it will add all files.
git add --all is another way to add all files.

> git commit

  • Committing puts the staged files into the repository
    If files have changed, a diff is created reflecting the change.
  • Committing is like taking a snapshot of the code
    In the Version Control section, I discussed creating new folders to keep track of versions.
    Committing is more or less the equivalent, but a much cleaner solution.
  • The commit message is used to document changes
    git commit -m 'Make some really cool changes'
    The option -m 'msg' is a shortcut to add all tracked files and write a commit message in line before committing.
Commit often and write detailed messages, allows easier fault finding and collaboration.

The .gitignore file

  • You don't want all files to be tracked in your repository
    Remember those binary files? They shouldn't be tracked.
    Local config for the project, IDEs and OS specific files should also be excluded.
  • gitignore.io
    This is a great resource to create a .gitignore file for your project.
Talk about 200MB zip file fiasco at work.

> git log

  • Shows the history of the repo
    This is an easy way to see what changes have been made recently.
  • Each commit in the log contains:
    - Who wrote it
    - When they committed it
    - A hash to uniquely identify it
    - The commit message

> git branch <branchname>

  • Branches are different versions of a repo
    By default a repo is on the master branch, this ideally should be the stable build of the project.
    When we're developing something, we're going to break it. Having a safe space to do this is what branches provide.
  • git branch feature/hello-nasser
    This creates a branch identical to the current branch for us to experiment on.

> git checkout <branch|commit>

  • Allows us to change branches at any time
    git checkout feature/hello-nasser
    This takes us to the feature/hello-nasser branch so we can make some changes without affecting others.
  • Shortcut to create a branch and check it out
    git checkout -b <branch>
  • Also allows us to go to any point in the history of the repo
    Looking at the files after checking out a previous commit shows that we've gone back in time.

> git merge <branchname>

  • Merges changes from one branch into another
    Once we've finished with out changes to the feature/hello-nasser branch, we'll want to propagate those changes to the main branch.
    To do this, we checkout the master branch then merge changes from the feature/hello-nasser branch.
    git checkout master
    git merge feature/hello-nasser
  • Merge Conflicts
    If two people change the same lines of one file, the merge algorithm git uses isn't smart enough to tell which branch contains the correct changes. The responsibility falls to you to sort out the conflict.

Chapter 3

GitHub

What is Github?

GitHub is a web app that hosts your repos remotely.

It has a great web interface to see commits and statistics.

It allows you to:

  • Backup your code
  • Collaborate with others easily
  • Have a 'true' source of code

Centralised vs. Distributed VCS

  • Centralised
    No files need to be kept on the programmer's HDD
    All changes made need to be committed to the central repo
  • Distributed
    Each person working on the code has a complete clone of the repo and contains all metadata.
    No dependence on a single source, but often there is a 'origin' for easy collaboration.
    Git is a distributed and GitHub is often used as the 'true source' remote repo.

Creating a Repo

  • Name
    • Identifier on GitHub
    • Used as default directory name when cloning the repo.
  • Public/Private
    • Public means that other GitHub users can browse and, download and fork your repo.
    • This encourages open source software development!
    • Public repos are hosted for free :D
    • Private repos are invisible to everyone except owner and collaborations.
    • You can invite people to collaborate on your project.
    • Student accounts have unlimited free private repos!
    • Use these for assignments and personal projects. Make them public after marks are released.
    • Having your code public on github is a great way to show potential employers your skills.

Connecting to GitHub

  • Add the GitHub repo as the 'origin' remote repo
    • git remote add origin git@github.com:drpotato/git-lecture.git
    • You can have any number of remotes. Origin is usually the 'upstream' or 'source of truth'
    • Remotes can be any type of url, even a local file system url
  • Get your code on GitHub
    • git push <remote> <branch>
    • add -u option to set branch as upstream.

Pull Requests

When we've made some changes to the code, but don't have sufficient permission to write data, you'll need to make a pull request to the owner of the repo.
  • What is it?
    • Sends a request to the owner to pull (merge) the changes you made into their repo.
    • This allows collaborators to view and discuss changes in a branch as they are made.
    • Developers can comment on pull requests, even on specific lines of code.

Issues

  • What are issues?
    • Issues have the advantage over comments that they can be referenced in commit messages (you can also reference commit numbers eg “from b4ebbd4”)
    • e.g. git commit -m 'Change countdown so it actually stops - fix #2'
    • A finished PR can be merged back into master. This will close all the issues that have been fixed in that branch and put the code back into master.

Pulling Changes

  • Changes made on a remote branch need to be fetched
    • git pull origin master
    • This fetches changes from the origin repo and merges them
  • git pull is actually a shortcut
    • git fetch <remote> <branch>
    • git merge FETCH_HEAD
    • The pull shortcut is a lot easier, you'll probably never fetch manually.
Always pull changes before you start working on something. Talk about Tomy deleting all changes due to merge conflits.

Cloning a Repo

Downloads the repo to your local machine

  • git clone [url] <directory>
    • <directory> is optional, by default it's the repo at the end of [url]
    • This fetches changes from the origin repo and merges them
    • Automatically uses [url] as origin

Undoing things

This is dangerous stuff...

But fortune favours the bold!

  • git reset [filename]
    Unstages the file.
  • git checkout [filename]
    Resets (permanently) a file to its state at the last commit.
    You can checkout any version of a file, read the docs to figure it out.
  • git commit --amend
    Includes all changes since the last commit into the last commit.
    Useful for when you forget to add a file.
  • git reset --hard HEAD
    Throws away (permanently) all changes since the last commit.

Seeing What Changed

How do we see what has changed between versions?

  • git diff
    Shows all changes to files since the last commit.
  • git diff [filename]
    Shows all changes to a specific file since the last commit.
  • git diff [commit]
    Shows all changes since the commit.
  • git diff <branch>..<branch>
    Shows the difference between two branches.

Recap

What you should take away

  • The basics of working with Git
    • git init
    • git add .
    • git commit -m 'message'
  • Looking at your repo
    • git status
    • git log
    • git diff
  • Avoid graphical clients!
    "Why?" you might ask:
    • Easier to make mistakes
    • Great opportunity to learn command line tools
    • Often leads to messy repo with many untracked files
